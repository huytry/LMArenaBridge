🚀 LMArena Bridge - The Next-Generation OpenAI Bridge 🌉
Welcome to the new generation of LMArena Bridge! 🎉 This is a high-performance toolset based on FastAPI and WebSocket that allows you to seamlessly use the vast number of large language models available on LMArena.ai through any client or application compatible with the OpenAI API.

This refactored version aims to provide a more stable, maintainable, and extensible experience.

✨ Main Features
🚀 High-Performance Backend: Based on FastAPI and Uvicorn, providing asynchronous, high-performance API services.

🔌 Stable WebSocket Communication: Uses WebSocket instead of Server-Sent Events (SSE) for more reliable, low-latency, bidirectional communication.

🤖 OpenAI-Compatible Interface: Fully compatible with OpenAI v1/chat/completions, v1/models, and v1/images/generations endpoints.

📋 Manual Model List Update: A new model_updater.py script allows you to manually trigger the extraction of the latest available model list from the LMArena page, saving it as available_models.json for easy reference and updating of the core models.json.

📎 Universal File Upload: Supports uploading any file type (images, audio, PDFs, code, etc.) via Base64, including support for multiple files in a single upload.

🎨 Text-to-Image Support: Added text-to-image functionality, allowing you to call LMArena's image generation models through the standard OpenAI interface.

🗣️ Full Conversation History Support: Automatically injects conversation history into LMArena for contextual, continuous dialogue.

🌊 Real-time Streaming Responses: Receive text responses from models in real-time, just like the native OpenAI API.

🔄 Automatic Program Updates: Automatically checks the GitHub repository on startup and can download and apply updates when a new version is found.

🆔 One-Click Session ID Update: The id_updater.py script allows you to automatically capture and update the required session ID in config.jsonc with just one action in your browser.

⚙️ Browser Automation: The accompanying Tampermonkey script (LMArenaApiBridge.js) is responsible for communicating with the backend server and performing all necessary actions in the browser.

🍻 Tavern Mode: Specially designed for applications like SillyTavern, it intelligently merges system prompts to ensure compatibility.

🤫 Bypass Mode: Attempts to bypass the platform's content filters by injecting an extra empty user message into the request.

🔐 API Key Protection: You can set an API Key in the configuration file to add a layer of security to your service.

🎯 Advanced Model-Session Mapping: Supports configuring independent session ID pools for different models and specifying a working mode (e.g., battle or direct_chat) for each session, enabling more granular request control.

⚙️ Configuration File Explanation
The project's main behavior is controlled by config.jsonc, models.json, and model_endpoint_map.json.

models.json - Core Model Mapping
This file contains the mapping from model names on the LMArena platform to their internal IDs.

Important: This is a required core file for the program to run. You need to manually maintain this list to ensure it includes the models you want to use via the API.

When a client requests a model, api_server.py looks up the corresponding model ID in this file.

available_models.json - Available Models Reference (Optional)
This is a reference file generated by the new model_updater.py script.

It contains the complete information (ID, name, organization, etc.) for all models extracted from the LMArena page.

You can run model_updater.py to generate or update this file, then copy the model information you need from it into models.json.

config.jsonc - Global Configuration
This is the main configuration file, containing global settings for the server.

session_id / message_id: The global default session IDs. These are used when a specific mapping for a model is not found in model_endpoint_map.json.

id_updater_last_mode / id_updater_battle_target: The global default request mode. Similarly, this setting is used when a specific session does not have a mode specified.

use_default_ids_if_mapping_not_found: A very important switch (defaults to true).

true: If the requested model is not found in model_endpoint_map.json, the global default IDs and mode will be used.

false: If no mapping is found, an error will be returned directly. This is useful when you need to strictly control the session for each model.

For other configuration items like api_key, tavern_mode_enabled, etc., please refer to the comments within the file.

model_endpoint_map.json - Model-Specific Configuration
This is a powerful advanced feature that allows you to override global settings and configure one or more dedicated sessions for specific models.

Core Advantages:

Session Isolation: Use separate sessions for different models to prevent context leakage.

Increased Concurrency: Configure an ID pool for popular models. The program will randomly select an ID for each request, simulating round-robin to reduce the risk of a single session being requested too frequently.

Mode Binding: Bind a session ID to the mode it was captured in (direct_chat or battle) to ensure the request format is always correct.

Configuration Example:

{
  "claude-3-opus-20240229": [
    {
      "session_id": "session_for_direct_chat_1",
      "message_id": "message_for_direct_chat_1",
      "mode": "direct_chat"
    },
    {
      "session_id": "session_for_battle_A",
      "message_id": "message_for_battle_A",
      "mode": "battle",
      "battle_target": "A"
    }
  ],
  "gemini-1.5-pro-20241022": {
      "session_id": "single_session_id_no_mode",
      "message_id": "single_message_id_no_mode"
  }
}

Opus: An ID pool is configured. A random session will be chosen upon request, and the request will be sent strictly according to its bound mode and battle_target.

Gemini: Uses a single ID object (old format, still compatible). Since it doesn't specify a mode, the program will automatically use the global mode defined in config.jsonc.

🛠️ Installation and Usage
You will need a Python environment and a browser that supports Tampermonkey scripts (e.g., Chrome, Firefox, Edge).

1. Prerequisites
Install Python Dependencies Open a terminal, navigate to the project root directory, and run the following command:

pip install -r requirements.txt

Install a Tampermonkey Script Manager Install the Tampermonkey extension for your browser.

Install the Project's Tampermonkey Script

Open the Tampermonkey extension's dashboard.

Click "Create a new script".

Copy and paste all the code from the TampermonkeyScript/LMArenaApiBridge.js file into the editor.

Save the script.

2. Run the Main Program
Start the Local Server In the project root directory, run the main service program:

python api_server.py

When you see the server start message on http://127.0.0.1:5102, it means the server is ready.

Keep the LMArena Page Open Ensure you have at least one LMArena page open and that the Tampermonkey script has successfully connected to the local server (the page title will start with a ✅). You don't need to stay on a conversation page; any page under the domain, like the Leaderboard, will work.

3. Update the Available Model List (Optional, but Recommended)
This step generates the available_models.json file, letting you know which models are currently available on LMArena so you can update models.json.

Ensure the main server is running.

Open a new terminal and run the model updater:

python model_updater.py

The script will automatically request the browser to fetch the model list and generate the available_models.json file in the root directory.

Open available_models.json, find the models you want, and copy their "publicName" and "id" key-value pairs into the models.json file (in the format "publicName": "id").

4. Configure Session ID (Do this once, or when switching models or if the session becomes invalid)
This is the most important step. You need to obtain a valid session ID and message ID so the program can communicate correctly with the LMArena API.

Ensure the Main Server is Running api_server.py must be running because the ID updater needs it to activate the capture function in the browser.

Run the ID Updater Open a new terminal and run the id_updater.py script in the project root directory:

python id_updater.py

The script will prompt you to choose a mode (DirectChat / Battle).

After selection, it will notify the running main server.

Activate and Capture

At this point, you should see a crosshair icon (🎯) appear at the beginning of the LMArena page title in your browser. This indicates that ID capture mode is active.

In the browser, open a page where the target model has sent a message. Note: If it's a Battle page, do not view the model names to remain anonymous, and ensure the last message in the interface is a response from the target model. If it's a Direct Chat, ensure the last message is a response from the target model.

Click the Retry button in the upper-right corner of the target model's response card.

The Tampermonkey script will capture the sessionId and messageId and send them to id_updater.py.

Verify the Result

Go back to the terminal where you ran id_updater.py. You will see that it has printed the successfully captured IDs and confirmed they have been written to the config.jsonc file.

The script will automatically close upon success. Your configuration is now complete!

5. Configure Your OpenAI Client
Point your client or application's OpenAI API address to the local server:

API Base URL: http://127.0.0.1:5102/v1

API Key: If api_key in config.jsonc is empty, you can enter anything. If it is set, you must provide the correct key.

Model Name: Specify the name of the model you want to use in your client ( must exactly match a name in models.json). The server will look up the corresponding model ID based on this name.

6. Start Chatting! 💬
You can now use your client normally. All requests will be proxied through the local server to LMArena!

🤔 How Does It Work?
This project consists of two parts: a local Python FastAPI server and a Tampermonkey script running in your browser. They work together via WebSocket.

sequenceDiagram
    participant C as OpenAI Client 💻
    participant S as Local FastAPI Server 🐍
    participant MU as Model Updater (model_updater.py) 📋
    participant IU as ID Updater (id_updater.py) 🆔
    participant T as Tampermonkey Script 🐵 (on LMArena page)
    participant L as LMArena.ai 🌐

    alt Initialization
        T->>+S: (Page Load) Establish WebSocket connection
        S-->>-T: Acknowledge connection
    end

    alt Manual Model List Update (Optional)
        MU->>+S: (User runs) POST /internal/request_model_update
        S->>T: (WebSocket) Send 'send_page_source' command
        T->>T: Scrape page HTML
        T->>S: (HTTP) POST /internal/update_available_models (with HTML)
        S->>S: Parse HTML and save to available_models.json
        S-->>-MU: Acknowledge
    end

    alt Manual Session ID Update
        IU->>+S: (User runs) POST /internal/start_id_capture
        S->>T: (WebSocket) Send 'activate_id_capture' command
        T->>L: (User clicks Retry) Intercept fetch request
        T->>IU: (HTTP) Send captured IDs
        IU->>IU: Update config.jsonc
        IU-->>-T: Acknowledge
    end

    alt Normal Chat Flow
        C->>+S: (User chats) /v1/chat/completions request
        S->>S: Convert request to LMArena format (get model ID from models.json)
        S->>T: (WebSocket) Send message with request_id and payload
        T->>L: (fetch) Send the actual request to LMArena API
        L-->>T: (Stream) Return model response
        T->>S: (WebSocket) Send response chunks back
        S-->>-C: (Stream) Return OpenAI-formatted response
    end

    alt Text-to-Image Flow
        C->>+S: (User requests) /v1/images/generations request
        S->>S: (In parallel) Create n tasks
        S->>T: (WebSocket) Send n tasks with request_ids
        T->>L: (fetch) Send n actual requests
        L-->>T: (Stream) Return image URLs
        T->>S: (WebSocket) Send URLs back
        S-->>-C: (HTTP) Return JSON response with all URLs
    end

Establish Connection: When you open an LMArena page in your browser, the Tampermonkey script immediately establishes a persistent WebSocket connection with the local FastAPI server.

Note: The current architecture assumes only one browser tab is active. If multiple pages are open, only the last connection will be effective.

Receive Request: The OpenAI client sends a standard chat request to the local server, specifying the model name in the request body.

Task Distribution: After receiving the request, the server looks up the corresponding model ID from models.json based on the model name. It then converts the request into the format required by LMArena, attaches a unique request ID (request_id), and sends this task to the connected Tampermonkey script via WebSocket.

Execution and Response: Once the Tampermonkey script receives the task, it directly initiates a fetch request to LMArena's API endpoint. When LMArena returns a streaming response, the script captures these data chunks and sends them back to the local server piece by piece via WebSocket.

Response Relay: The server uses the request_id attached to each data chunk to place it in the correct response queue and streams this data back to the OpenAI client in real-time.

📖 API Endpoints
Get Model List
Endpoint: GET /v1/models

Description: Returns an OpenAI-compatible list of models read from the models.json file.

Chat Completions
Endpoint: POST /v1/chat/completions

Description: Receives standard OpenAI chat requests, supporting both streaming and non-streaming responses.

Image Generation
Endpoint: POST /v1/images/generations

Description: Receives standard OpenAI text-to-image requests and returns the generated image URLs.

Request Example:

curl http://127.0.0.1:5102/v1/images/generations \
  -H "Content-Type: application/json" \
  -d '{
    "prompt": "A futuristic cityscape at sunset, neon lights, flying cars",
    "n": 2,
    "model": "dall-e-3"
  }'

Response Example:

{
  "created": 1677663338,
  "data": [
    {
      "url": "https://..."
    },
    {
      "url": "https://..."
    }
  ]
}
